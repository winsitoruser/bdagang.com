================================================================================
EXECUTIVE SUMMARY
================================================================================

PARTNER (MITRA):
- External business entities yang menggunakan sistem
- Memiliki multiple outlets (cabang/toko)
- Requires activation approval process
- Subscription-based model
- B2B relationship dengan platform

TENANT:
- Internal system workspace/organization
- Core operational unit dalam sistem
- Module-based access control
- Direct admin creation (no approval needed)
- Multi-branch capability dengan HQ system

KEY DIFFERENCE:
Partner = Customer/Client yang membeli layanan
Tenant = Workspace/Organization yang menjalankan operasional

================================================================================
RELATIONSHIPS & ASSOCIATIONS
================================================================================

PARTNER RELATIONSHIPS:
Partner → PartnerOutlet (one-to-many)
  - Partner has many outlets/branches
  - Each outlet is a physical location

Partner → PartnerUser (one-to-many)
  - Partner has many users
  - Separate user table for partner users

Partner → PartnerSubscription (one-to-many)
  - Partner has subscription history
  - Tracks subscription packages

Partner → ActivationRequest (one-to-many)
  - Partner can have multiple activation requests
  - Tracks approval workflow

TENANT RELATIONSHIPS:
Tenant → BusinessType (many-to-one)
  - Tenant belongs to a business type
  - Determines default modules

Tenant → User (one-to-many)
  - Tenant has many users
  - Uses main User table with tenantId

Tenant → Module (many-to-many through TenantModule)
  - Tenant has enabled modules
  - Module-based feature access

Tenant → Branch (one-to-many)
  - Tenant has multiple branches
  - HQ-based multi-branch system

Tenant → Store (one-to-many)
  - Tenant has stores
  - Physical store locations

Tenant → SyncLog (one-to-many)
  - Tenant has sync logs
  - Tracks data synchronization

================================================================================
BUSINESS LOGIC DIFFERENCES
================================================================================

PARTNER WORKFLOW:
1. Partner Registration
   ├─ Partner submits registration form
   ├─ Status: pending
   └─ ActivationRequest created

2. Admin Review
   ├─ Admin reviews partner application
   ├─ Can approve or reject
   └─ Sets activation_status

3. Approval
   ├─ If approved: status → active
   ├─ Partner can access system
   └─ Can create outlets

4. Subscription
   ├─ Partner subscribes to package
   ├─ PartnerSubscription created
   └─ Billing/payment tracking

5. Operations
   ├─ Partner manages outlets
   ├─ Partner users access system
   └─ Transaction tracking per outlet

TENANT WORKFLOW:
1. Tenant Creation (Admin)
   ├─ Admin creates tenant directly
   ├─ Selects business type
   ├─ Selects subscription plan
   └─ Status: trial

2. Owner User Creation
   ├─ System auto-creates owner user
   ├─ Generates temp password
   └─ Owner receives credentials

3. Module Assignment
   ├─ Default modules auto-enabled
   ├─ Based on business type
   └─ TenantModule records created

4. Onboarding
   ├─ Owner logs in
   ├─ Completes 5-step wizard
   └─ setup_completed = true

5. Operations
   ├─ Tenant users access modules
   ├─ Can add branches (if multi-branch)
   ├─ Full ERP functionality
   └─ Module-based access control

================================================================================
USE CASES & SCENARIOS
================================================================================

WHEN TO USE PARTNER:
✓ B2B SaaS model where businesses sign up as customers
✓ Need approval workflow before activation
✓ Multiple outlets/locations per customer
✓ Subscription package management
✓ External customer relationship
✓ Separate user management for customers
✓ Partner portal/dashboard needed

EXAMPLE:
- Franchise management system
- Multi-outlet retail chains signing up
- Restaurant chains as customers
- Third-party businesses using platform

WHEN TO USE TENANT:
✓ Internal workspace/organization model
✓ Direct admin-created accounts
✓ Module-based feature access
✓ Multi-branch with HQ synchronization
✓ Onboarding wizard for setup
✓ Full ERP system access
✓ Integrated user management

EXAMPLE:
- Single business using ERP
- Company with multiple branches
- Organization needing full system access
- Internal business operations

POSSIBLE SCENARIOS:

SCENARIO 1: SEPARATE SYSTEMS
- Partners: External B2B customers
- Tenants: Internal operational units
- No direct relationship between them

SCENARIO 2: PARTNER → TENANT CONVERSION
- Partner signs up (external)
- After approval, Tenant created (internal workspace)
- Partner.id could link to Tenant (currently NOT implemented)

SCENARIO 3: HYBRID MODEL
- Some businesses are Partners (B2B customers)
- Some businesses are Tenants (direct ERP users)
- Different pricing/access models

================================================================================
RECOMMENDATION FOR CLARITY
================================================================================

CURRENT STATE:
The system has BOTH Partner and Tenant models but NO direct relationship
between them. This suggests they serve different purposes:

PARTNER = Customer/Client Entity
- External relationship
- Requires approval
- Subscription-based
- Multiple outlets

TENANT = Workspace/Organization
- Internal operational unit
- Direct creation
- Module-based access
- Full ERP functionality

RECOMMENDED CLARIFICATION:

OPTION A: MERGE MODELS
If Partner and Tenant serve the same purpose:
- Consolidate into single model
- Add fields for both use cases
- Simplify architecture

OPTION B: CLEAR SEPARATION
If they serve different purposes:
- Partner = B2B customer entity
- Tenant = ERP workspace
- Add relationship: Partner → Tenant (one-to-one or one-to-many)
- Partner creates/owns Tenant after approval

OPTION C: CURRENT STATE (Keep Both)
Maintain dual model for flexibility:
- Partner for external customer management
- Tenant for internal ERP operations
- No direct link (different use cases)

CONCLUSION
================================================================================

PARTNER (MITRA):
- Represents external business customers
- Requires approval before activation
- Has outlets, users, and subscriptions
- More like a "client" or "customer" entity
- B2B relationship model

TENANT:
- Represents internal workspace/organization
- Direct admin creation (no approval)
- Module-based feature access
- Full ERP operational unit
- Core system entity for business operations

KEY INSIGHT:
These are TWO DIFFERENT CONCEPTS serving different purposes:
- Partner = Who is using the platform (customer)
- Tenant = What workspace they're using (operational unit)

POTENTIAL RELATIONSHIP:
Partner (approved) → Creates/Owns → Tenant (workspace)
Currently NOT implemented, but could be added if needed.
